# Context Engine MCP — V3

> **Goal:** Full automation layer. The system acts proactively — not just when asked.  
> **Requires:** V1 (Memory + RAG) + V2 (Analysis + Planning + Generation) fully working.

---

## 🎯 V3 Goal

Move from **reactive** (user asks → system answers) to **proactive** (system detects → system acts):

- Emails automatically become tasks in memory
- Events trigger workflows without manual input
- Team members share a collective memory
- New tools can be added without touching core code

**V3 success criteria:**
> An email arrives → system reads it → extracts tasks → adds to memory → notifies user. Zero manual steps.

## 📦 Module 1 — Workflow / Automation Module

### What it does

Event-driven pipelines. Define trigger → action rules that run automatically.

### Concepts

```
Trigger    →    Condition    →    Action
────────────────────────────────────────────
new email       from: client      extract tasks + add to memory
new entry       type: decision    notify via webhook
daily 09:00     -                 generate standup summary
weekly friday   -                 generate week summary post
entry updated   status: done      log to standup
```

### MCP Tools

```ts
create_workflow(input: {
  name: string
  trigger: {
    type: 'schedule' | 'entry_created' | 'entry_updated' | 'email_received' | 'webhook'
    config: object    // cron string, entry type filter, email filter, etc.
  }
  condition?: {
    field: string
    operator: string
    value: string
  }
  actions: {
    type: 'add_entry' | 'update_entry' | 'generate_post' | 'send_webhook' | 'run_agent'
    config: object
  }[]
}) → Workflow
```

```ts
list_workflows(input: { active?: boolean }) → Workflow[]
toggle_workflow(input: { id: string, active: boolean }) → Workflow
run_workflow(input: { id: string }) → WorkflowRun
get_workflow_runs(input: { workflow_id?: string, limit?: number }) → WorkflowRun[]
```

### Example Workflows (built-in presets)

```ts
// Daily standup summary
{
  trigger: { type: 'schedule', config: { cron: '0 9 * * 1-5' } },
  actions: [{ type: 'run_agent', config: { goal: 'generate standup from yesterday entries' } }]
}

// Weekly LinkedIn post
{
  trigger: { type: 'schedule', config: { cron: '0 8 * * 5' } },
  actions: [{ type: 'generate_post', config: { focus: 'my recent work', platform: 'linkedin' } }]
}

// Decision made → log note
{
  trigger: { type: 'entry_created', config: { type: 'decision' } },
  actions: [{ type: 'add_entry', config: { type: 'log', title: 'Decision logged: {{title}}' } }]
}
```

### Web UI — Workflow Page

```
/workflows
  ├── List of active workflows (toggle on/off)
  ├── [+ New Workflow] → visual builder
  │     ├── Pick trigger (schedule / event)
  │     ├── Add condition (optional)
  │     └── Add actions
  ├── Run history per workflow
  └── Preset templates (standup, weekly post, etc.)
```

## 📦 Module 2 — Email Module

### What it does

- Connect email accounts (IMAP or Gmail API)
- Ingest emails periodically (or on webhook)
- Summarize email content
- Extract tasks, decisions, follow-ups automatically
- Link emails to existing projects / people in memory

### MCP Tools

```ts
connect_email(input: {
  provider: 'gmail' | 'imap'
  config: object            // OAuth tokens or IMAP credentials
}) → EmailAccount
```

```ts
fetch_emails(input: {
  account_id: string
  since?: string            // ISO date
  limit?: number            // default: 20
  filter?: {
    from?: string
    subject_contains?: string
    unread_only?: boolean
  }
}) → Email[]
```

```ts
process_email(input: {
  email_id: string
  extract_tasks?: boolean   // default: true
  link_project?: string     // auto-link to project
  link_person?: string      // auto-link to person
}) → {
  summary: string
  tasks_created: Entry[]
  decisions_found: Entry[]
  follow_ups: Entry[]
}
```

```ts
list_email_accounts() → EmailAccount[]
get_email_summary(input: { since?: string }) → { total, processed, tasks_created }
```

### Auto-processing Flow

```
1. Email arrives (webhook or polling)
2. fetch_emails() → get unread
3. For each email:
   → summarize content
   → RAG search: find related project/person in memory
   → extract tasks, decisions, follow-ups
   → add_entry() for each extracted item
   → mark email as processed
4. User sees new tasks in Web UI / memory
```

## 📦 Module 3 — Team Memory

### What it does

Multiple users share a collective memory space. Each user has private entries + access to shared entries.

### Concepts

```
Memory Spaces:
  personal    → only me
  shared      → all team members
  project     → all members of a project

Roles:
  owner       → full access
  member      → read + write shared
  viewer      → read only
```

### Schema additions

```sql
create table teams (
  id          uuid primary key default gen_random_uuid(),
  name        text not null,
  created_at  timestamptz default now()
);

create table team_members (
  team_id     uuid references teams(id),
  user_id     uuid,
  role        text default 'member',  -- 'owner' | 'member' | 'viewer'
  primary key (team_id, user_id)
);

-- Add to entries:
alter table entries add column visibility text default 'personal';
-- 'personal' | 'team' | 'project'
alter table entries add column team_id uuid references teams(id);
```

### MCP Tools

```ts
create_team(input: { name: string, members: string[] }) → Team
invite_member(input: { team_id: string, email: string, role: string }) → TeamMember
get_team_memory(input: { team_id: string, type?: string, since?: string }) → Entry[]
```

## 📦 Module 4 — Plugin System

### What it does

Add new MCP tools without modifying core code. Each plugin is a self-contained module.

### Plugin Structure

```ts
// A plugin is just an object:
interface Plugin {
  name: string
  version: string
  tools: MCPTool[]          // new tools to register
  schema?: SQLMigration[]   // optional new DB tables
  setup?: () => Promise<void>
}
```

### Plugin Registry

```
/plugins
  /github-integration       ← sync PRs, issues to memory
  /notion-import            ← import Notion pages as entries
  /slack-integration        ← Slack messages → tasks
  /custom                   ← user-built plugins
```

### MCP Tools

```ts
list_plugins() → Plugin[]
install_plugin(input: { source: string }) → Plugin  // npm package or local path
enable_plugin(input: { name: string }) → void
disable_plugin(input: { name: string }) → void
```

## ✅ Decisions Made

| Decision | Choice | Why |
|----------|--------|-----|
| Workflows | Event-driven + schedule | Cover both reactive and proactive cases |
| Email | Gmail API primary, IMAP fallback | Gmail = most common, IMAP = flexibility |
| Team memory | Visibility field on entries | Simple, no duplicate schema |
| Plugins | npm packages or local path | Ecosystem-friendly, easy to share |
| Auth | Supabase Auth | Already using Supabase, zero extra infra |
