# Context Engine MCP — V2

> **Goal:** Add intelligence on top of V1 memory. Analyze code, plan tasks, and generate content — all driven by Claude Code / Copilot CLI, with results saved to Supabase and visible in the app.  
> **Requires:** V1 fully working (Memory + RAG + Web UI)

---

## 🎯 V2 Goal

Three new modules that use V1 memory as their foundation:

1. **Analysis Module** — Claude/Copilot analyzes the codebase → saves insights to Supabase → visible in app
2. **Planning Module** — Claude/Copilot generates task structure → saved via MCP → user manages automation in app
3. **Generation Module** — files uploaded to Supabase Storage → RAG searches across storage + memory

**V2 architectural principle:**
> The CLI AI (Claude Code / Copilot) is the reasoning engine.  
> MCP tools are the write interface.  
> Supabase is the single source of truth.  
> The web app is the read/manage surface.

**V2 success criteria:**
> Start a new chat → AI automatically loads rules from Supabase and behaves accordingly.  
> Run an analysis session → insights appear in the app automatically.  
> Plan a goal in the CLI → task breakdown is visible and actionable in the app.  
> Upload files to Supabase Storage → RAG can surface content from those files alongside memory entries.

---

## 🧱 Additional Stack (V2)

| Layer | Technology | Why |
|-------|-----------|-----|
| File Storage | Supabase Storage (bucket) | Files stored remotely, accessible by RAG + app |
| Web Search | Brave Search API (or Tavily) | Real-time news + internet for Generation |
| Scraping/Fetch | Native fetch + cheerio | Parse news articles |

> No local `fs` reads — files live in Supabase Storage. Claude/Copilot CLI handles all reasoning; MCP tools handle writes to Supabase.

## 📦 Module 0 — Rules Engine *(foundation for all modules)*

### What it does

- Stores AI usage rules in Supabase (`rules` table)
- On every new chat/session, the AI automatically loads active rules via MCP
- Rules are editable in the app without touching code
- Rules are injected into the AI's context before any work begins

### How it works — session init

```
1. Chat starts
   → AI (via agent .md instructions) calls: get_context()
   → get_context() now includes: system_rules[] from Supabase
   → AI reads the rules and applies them for the rest of the session

2. User edits rules in app (/rules page)
   → saved to Supabase immediately
   → next session automatically picks up the changes
```

### DB Schema

```sql
create table rules (
  id          uuid primary key default gen_random_uuid(),
  title       text not null,         -- short label, e.g. "Language"
  content     text not null,         -- the actual rule, e.g. "Always respond in PT"
  category    text,                  -- 'behavior' | 'memory' | 'output' | 'general'
  active      boolean default true,
  priority    int default 0,         -- higher = injected first
  created_at  timestamptz default now(),
  updated_at  timestamptz default now()
);
```

### MCP — extend `get_context`

`get_context` already exists in V1. Extend it to include rules:

```ts
get_context(input: { question: string }) → {
  context: string           // formatted RAG results (existing)
  system_rules: {           // NEW — active rules from Supabase
    title: string
    content: string
    category: string
  }[]
}
```

The agent `.md` file gets one new instruction:
```
⚡ SESSION START: call get_context({ question: "session init" }) before anything else.
   Apply all returned system_rules for this entire session.
```

### New MCP tools

```ts
get_rules(input: {
  category?: string   // filter by category
  active?: boolean    // default: true
}) → Rule[]

add_rule(input: {
  title: string
  content: string
  category?: string
  priority?: number
}) → Rule

update_rule(input: {
  id: string
  title?: string
  content?: string
  active?: boolean
  priority?: number
}) → Rule

delete_rule(input: { id: string }) → void
```

### Web UI — `/rules` page

```

  ├── Header: "AI Rules" + [+ Add Rule] button
  │
  ├── Filter tabs: All | Behavior | Memory | Output | General
  │
  ├── Rule cards (sorted by priority):
```

## 📦 Module 1 — Analysis Module

### What it does

- **Claude Code / Copilot CLI** analyzes the codebase (the AI has file access natively)
- Uses MCP tool `save_analysis` to persist insights → Supabase
- Insights are immediately visible in the app (entries of type `analysis`)
- RAG can surface past analysis alongside memory entries

### How it works

```
1. User asks Claude Code: "analyze wizpress for tech debt"
   → Claude reads files natively (it's the CLI — it has access)
   → Claude calls: save_analysis({ project, summary, insights, files_referenced })
   → MCP saves to Supabase as entry (type: 'analysis')

2. App /analyze page shows all saved analyses
   → filterable by project, date, focus
   → each entry links to the files referenced
```

### MCP Tools

```ts
save_analysis(input: {
  project: string           // project name
  summary: string           // high-level summary of what was found
  insights: string          // detailed findings (tech debt, patterns, risks)
  focus?: string            // 'debt' | 'patterns' | 'dependencies' | 'general'
  files_referenced?: string[] // file paths Claude analyzed
  tags?: string[]
}) → Entry  // saved as type: 'analysis'
```

```ts
get_analyses(input: {
  project?: string
  focus?: string
  since?: string
  limit?: number
}) → Entry[]
```

> **No MCP file-reading tool needed.** Claude Code / Copilot already has native file access.  
> The MCP role is write-only: persist Claude's output to Supabase.

## 📦 Module 2 — Planning Module

### What it does

- **Claude Code / Copilot CLI** generates the task breakdown (AI reasons over goal + memory context)
- Uses MCP tool `save_plan` to persist the structure → Supabase
- Plan appears in the app as a collection of tasks (entries of type `task`)
- User manages and triggers automation from the app — AI doesn't auto-execute

### How it works

```
1. User asks Claude Code: "plan adding auth to kontext"
   → Claude calls: query_context({ question: "kontext architecture decisions stack" })
   → Gets relevant memory context
   → Reasons over it → generates task breakdown
   → Calls: save_plan({ goal, project, tasks[] })
   → MCP saves each task as entry (type: 'task', status: 'pending')

2. App /plan page shows all plans + their tasks
   → User sees the breakdown
   → Can mark tasks done, reorder, trigger automation
   → suggest_next() surfaces what to do next (already in V1 via get_entries filters)
```

### MCP Tools

```ts
save_plan(input: {
  goal: string              // "add auth to kontext"
  project?: string
  tasks: {
    title: string
    description?: string
    tags?: string[]
    depends_on?: string     // title of predecessor task
  }[]
}) → {
  plan_entry: Entry         // parent entry (type: 'plan')
  tasks_created: Entry[]    // child entries (type: 'task', status: 'pending')
}
```

> **No auto-execution.** Claude generates structure, user decides what to automate.  
> `suggest_next` is already covered by `get_entries({ status: 'pending' })` from V1.

## 📦 Module 3 — Generation Module

### Concept

> The user defines **who they are** (rules + intent) and the system generates posts that combine **their real work** (memory) with **what's happening in the world** (internet/news).

Posts feel authentic because they ARE — they're based on real decisions, real work, real context.

### File Storage — Supabase Storage Bucket

Files (docs, screenshots, code snippets, notes) are uploaded to a **Supabase Storage bucket** instead of being read locally.

```
bucket: context-engine-files/
  ├── {user_id}/
  │     ├── documents/
  │     ├── code-snippets/
  │     └── notes/
```

RAG works across **both** memory entries (Supabase DB) and files in storage:
- Files are indexed on upload → embeddings stored in `embeddings` table
- `search_memory` searches across entries + file chunks transparently

### How it works

```
1. User uploads file to app (or via MCP tool upload_file)
   → stored in Supabase Storage bucket
   → file content extracted + chunked
   → embeddings generated + saved to embeddings table
   → entry created (type: 'file', metadata: { bucket_path, filename })

2. User asks Claude Code: "generate LinkedIn posts about my work this week"
   → Claude calls: generate_posts({ platform: 'linkedin', focus: 'mixed' })
   → MCP: RAG over memory + file chunks → recent work context
   → MCP: fetch_world_context → news on user's topics
   → Claude generates drafts using both sources
   → MCP: returns drafts to Claude
   → User reviews in CLI or app

3. User approves → calls save_post({ content, platform })
   → saved to Supabase as entry (type: 'post')
```

### MCP Tools

```ts
upload_file(input: {
  path: string              // local path to upload
  category?: string         // 'documents' | 'code-snippets' | 'notes'
  project?: string          // associate with a project
  tags?: string[]
}) → {
  entry: Entry              // type: 'file'
  bucket_path: string       // where it lives in Supabase Storage
  chunks_indexed: number    // how many chunks were embedded
}
```

```ts
set_generation_profile(input: {
  platform: 'linkedin' | 'twitter' | 'newsletter'
  intent: string        // "position myself as an AI engineer building autonomous systems"
  tone: string          // "technical but approachable, no buzzwords, first-person"
  topics: string[]      // ["AI agents", "developer tools", "MCP", "TypeScript"]
  avoid: string[]       // ["politics", "crypto", "AI doom/hype"]
  post_frequency?: string  // "2-3x per week"
  language?: string     // "english" | "portuguese" | "both"
}) → Profile
```

```ts
fetch_world_context(input: {
  topics: string[]          // pulled from profile if not provided
  since?: string            // 'today' | '3 days' | 'week'
  limit?: number            // articles to fetch (default: 10)
}) → {
  articles: { title, summary, url, published_at }[]
}
```

```ts
generate_posts(input: {
  platform?: string         // defaults to profile platform
  count?: number            // how many drafts (default: 3)
  focus?: string            // "my recent work" | "world news" | "mixed" (default: mixed)
  topic_hint?: string       // optional extra focus e.g. "focus on MCP launch"
}) → {
  posts: {
    content: string
    hook: string            // first line (most important for LinkedIn)
    topics_used: string[]
    memory_sources: Entry[] // which memory entries informed this
    news_sources: { title, url }[]
  }[]
}
```

```ts
save_post(input: {
  content: string
  platform: string
  published?: boolean
  published_at?: string
}) → Entry  // saved to memory as type: 'post'
```

## 💡 V2 Enhancements & Open Questions *(2026-04-01)*

### Features que faltam no scope atual

| Feature | Descrição |
|---|---|
| **Session history** | Cada sessão do Claude gravada como entidade `session` — título, duração, entries criadas, projeto ativo. Responde a "o que foi feito ontem" com real granularidade |
| **Entry relationships** | Links explícitos tipados entre entries: `decision → tasks` que derivaram dela, `analysis → decisions` gerada, `plan → tasks` filho. Não só wikilinks — foreign keys reais |
| **Memory health UI** | Dashboard de qualidade: duplicados detetados, entries órfãs (sem projeto), entries velhas sem status. Ação em bulk para limpar o ruído |
| **Context export** | Botão "Export as prompt" — gera um bloco formatado com rules ativas + entries relevantes. Colar em qualquer AI tool (ChatGPT, Gemini, etc.) e ter contexto completo |

### Features que tornam o produto mais vivo

| Feature | Descrição |
|---|---|
| **Staleness scoring** | Entries com +30 dias sem atualização marcadas como `stale`. Visível na Memory section como aviso passivo |
| **Smart nudges** | "3 decisions não revisadas há 2 semanas" — surface no Overview. Proativo mas sem ser automação full (isso é v3) |
| **RAG quality metrics** | Mostrar relevance score no Search por resultado. Ajuda a calibrar se o embedding está a funcionar bem |
| **MCP usage stats** | Contador de chamadas por tool + entries mais acedidas. Mostra que o sistema está a ser genuinamente usado pelo AI |

### Items over-scoped para v2 → mover para v3

| Item | Motivo |
|---|---|
| `fetch_world_context` (Brave/Tavily) | Dependência externa nova, edge cases de rate limit e scraping — melhor como trigger de workflow em v3 |
| **File Storage + RAG sobre ficheiros** | Chunking strategy, bucket management, e RAG híbrido (entries + chunks) é complexidade que merece módulo dedicado em v3 |

### Gaps técnicos a resolver

| Gap | Solução sugerida |
|---|---|
| **Realtime sem polling** | Definir canal Supabase Realtime por `user_id` (não por entry type) — evita N canais abertos |
| **Rules com prioridade igual** | Desempate por `created_at` (mais recente prevalece) + warning visual na UI quando há conflito |
| **`save_plan` com `depends_on` por título** | Aceitar tanto título como UUID — resolver por título em runtime, falhar com mensagem clara se ambíguo |

## ✅ Decisions Made

| Decision | Choice | Why |
|----------|--------|-----|
| Rules storage | Supabase `rules` table | Editable in app, no code changes needed |
| Rules injection | extend `get_context()` + session_rules[] | Reuses existing tool, zero new overhead |
| Rules load timing | Session start (agent .md instruction) | Guaranteed before any work begins |
| Analysis execution | Claude Code / Copilot CLI (native file access) | No MCP file-reading needed — CLI AI already reads files |
| Analysis persistence | `save_analysis` MCP tool → Supabase | Single source of truth, visible in app |
| Planning execution | Claude Code / Copilot CLI + `save_plan` MCP | AI reasons, user controls automation |
| Auto-execution | ❌ None — user manages from app | Safety, user control, no runaway agents |
| File storage | Supabase Storage bucket | Remote, accessible by RAG + app |
| RAG scope | Memory entries + file chunks (unified search) | Full context: notes + documents + code |
| Generation | Rules + intent + memory + file chunks + internet | Posts grounded in real work + real files |
| Web search | Brave Search API (or Tavily) | Real-time news without hallucinations |
| Social platform | LinkedIn primary | Professional audience, high value for positioning |
| Post drafts | Multiple drafts per run | User picks best, not locked into one |
| Posts storage | Saved as entries (type: 'post') | Reuses existing memory schema |
